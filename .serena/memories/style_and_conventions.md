# Style & conventions
- Backend: Python 3 (Dockerfile uses 3.11) with Flask blueprints wired in `backend/app/factory.py`. CORS is enabled globally, and custom error handlers (`app/errors.py`) convert HTTP exceptions and unforeseen failures into consistent JSON payloads (`{ error: { class,message,code,fields? } }`). `app/db.py` mimics Flask-SQLAlchemy and exposes `db.session`, `db.Model`, typed columns, and a helper `session_scope`. Models/jobs follow `snake_case` names, call `datetime.utcnow`/`now()` for timestamps, and prefer manual SQL+SQLAlchemy `text()` for job queue updates (`app/jobs/cli.py`). Audit logging (`app/audit.py`) writes to `audit_events` in a separate transaction and tolerates schema drift. Webhook posts (`app/webhook.py`) use standard library `urllib` + HMAC-SHA256, and CLI job handlers register via decorator-based registry in `app/jobs/handlers`.
- Frontend: Vite + React + TypeScript. Components are function components with hooks (`useState`, `useEffect`), `@mui/material` UI primitives, and module-scoped styles via the `sx` prop. API calls are centralized in `frontend/src/api.ts` with typed responses from `frontend/src/types.ts`, and environment configuration is grabbed from `import.meta.env.VITE_API_BASE`. Example components (e.g., `HSPage`) default form state, handle errors by peeling `e.response?.data?.error?.message`, and render tables with `<TableHead>/<TableBody>`.
- General habits: Japanese comments/docstrings are common in backend modules and docs, while code comments explain implementation details (e.g., why a scheduler retry uses SQL). JSON responses and API docs follow the OpenAPI schema in `openapi.yaml`, and there is little runtime type coercion—validators return 400/422 on missing payloads. Tests are not yet available, but documentation (`docs/implementation_plan_7_devex_quality.md`) calls for pytest + coverage in the future.
